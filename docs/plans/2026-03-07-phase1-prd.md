# Phase 1 PRD: Open-Source Schema + CLI Toolkit

**Product:** automagent — the open standard for defining AI agents
**Domain:** automagent.dev
**Phase:** 1 of 3
**Timeline:** 4 weeks
**Status:** Active

---

## 1. Vision

Automagent defines a portable, declarative, framework-neutral standard for AI agents. Phase 1 establishes the open-source standard and puts a working CLI in developers' hands — proving the schema has value before building platform infrastructure.

**Phase 1 mantra:** "The schema is the product. The CLI is the distribution. Everything else is earned after traction."

## 3. Scope

### In Scope (Phase 1)

1. **Lean monorepo structure** — pnpm workspaces with `packages/schema` and `packages/cli`
2. **CLI: `automagent init`** — interactive and `--quick` mode, generates valid agent.yaml with editor autocomplete
3. **CLI: `automagent validate`** — YAML parsing, JSON Schema validation, model recognition, API key detection, file reference checks
4. **CLI: `automagent run`** — local agent testing via model API, mock tool calls, interactive chat loop
5. **CLI: `automagent import`** — convert from CrewAI, OpenAI Agents SDK, and LangChain formats
6. **CLI: stub commands** — `push`, `pull`, `login`, `diff` print "Coming in Phase 2" with context
7. **npm publishing** — `@automagent/cli` scoped + `automagent` unscoped for `npx automagent`
8. **GitHub publishing** — public repo, README, contributing guidelines, issue templates
9. **Community launch materials** — launch post draft, design partner program doc

### Out of Scope (Phase 2+)

- Hosted registry (API, database, storage)
- Web UI
- Billing / pricing tiers
- GitHub Actions for CI publishing
- Automated benchmarking
- Framework translation (generating framework configs FROM agent.yaml)
- Marketplace features
- SSO / SCIM

## 4. User Personas

### Primary: Platform Engineer (Persona A)
> "My company has 15 teams building agents in 4 frameworks. I need to standardize."

- Has existing agents in CrewAI/OpenAI/LangChain
- Wants a single definition format that works across frameworks
- Will use `import` to convert existing agents, `validate` to enforce standards
- First to adopt, first to give schema feedback

### Secondary: AI/ML Engineer (Persona B)
> "I build agents weekly. I want to version them, test them, share them."

- Builds from scratch using `init`
- Uses `run` to iterate locally
- Wants the schema to capture everything about their agent in one file
- Provides feedback on missing fields and rough edges

### 5.3 CLI Command Specifications

#### `automagent init`
- **Interactive mode** (default): prompts for name, description, model, optional instructions
- **Quick mode** (`--quick`): uses args or sensible defaults
- **Output**: `agent.yaml` with `# yaml-language-server` directive for editor autocomplete
- **Guard**: refuses to overwrite existing file without `--force`
- **Post-create**: prints "Next steps" with validate/run commands

#### `automagent validate`
- **Input**: path to agent.yaml (defaults to `./agent.yaml`)
- **Pipeline**:
  1. Parse YAML (catch syntax errors with line numbers)
  2. Run JSON Schema validation via `@automagent/schema`
  3. Warn if model is unpinned alias (e.g., `gpt-4` without version)
  4. Error if high-entropy strings detected (potential API keys)
  5. Warn if file references in `context` don't exist on disk
- **Output**: checkmarks/warnings/errors with clear messages
- **Exit code**: 0 if valid (warnings OK), 1 if errors

#### `automagent run`
- **Input**: path to agent.yaml (defaults to `./agent.yaml`)
- **Behavior**:
  1. Parse and validate agent.yaml
  2. Detect model provider, check for API key in environment
  3. Initialize chat session with instructions as system prompt
  4. Mock tool calls by default (return placeholder responses)
  5. Interactive REPL: user types, agent responds
- **Flags**: `--model <override>` to test with different model, `--live` for real tool connections (future)
- **Error handling**: clear message if API key missing with exact env var name needed

#### `automagent import`
- **Input**: path to existing config file
- **Detection**: auto-detect format (CrewAI YAML, OpenAI JSON, LangChain)
- **Mappings**:
  - CrewAI: `role`→`name`, `goal`+`backstory`→`instructions`, `llm`→`model`, `tools`→`tools`
  - OpenAI: `name`→`name`, `instructions`→`instructions`, `model`→`model`, `tools`→`tools`, `handoffs`→`dependencies.agents`
  - LangChain: `prompt`→`instructions`, `model`→`model`, `tools`→`tools`
- **Output**: `agent.yaml` with `# TODO: Review` comments on unmapped fields
- **Post-import**: auto-runs validate on the generated file

#### Stub commands (`push`, `pull`, `login`, `diff`)
- Print a friendly message: "This command requires the automagent registry, launching in Phase 2."
- Include link to GitHub repo for updates
- Exit 0 (not an error)

## 6. Implementation Plan

### Step 1: Monorepo Restructure
- Create root `package.json` with pnpm workspaces
- Create `pnpm-workspace.yaml`
- Move `spec/` → `packages/schema/`
- Update internal references in CLAUDE.md and README
- Verify schema package still builds and tests pass

### Step 2: CLI Scaffold
- Create `packages/cli/` with Commander.js entry point
- Register all commands (init, validate, run, import + stubs)
- Set up build with tsup, bin entry for `automagent`
- Link `@automagent/schema` as workspace dependency
- Verify `automagent --help` and `automagent --version` work

### Step 3: Implement `init`
- Interactive prompts (name, description, model, instructions)
- `--quick` mode with CLI args
- YAML generation with schema directive
- Tests: quick mode, interactive mode, no-overwrite guard

### Step 4: Implement `validate`
- YAML parser with line-number error reporting
- Schema validation via `@automagent/schema`
- Warning checks (unpinned models, missing files, API key detection)
- Formatted output with colors
- Tests: valid file passes, missing fields fail, warnings shown

### Step 5: Implement `run`
- Agent runner: parse YAML, extract model + instructions
- Provider detection (anthropic/openai based on model string)
- Chat REPL with readline
- Tool mocker for declared tools
- Tests: YAML parsing, provider detection, graceful API key error

### Step 6: Implement `import`
- Format detection heuristics
- Three importers (CrewAI, OpenAI, LangChain)
- YAML output with TODO comments
- Auto-validate after import
- Tests: each framework maps correctly, unknown format errors helpfully

### Step 7: Polish + Publish
- Update root README with CLI documentation
- Stub commands with friendly messages
- Publish `@automagent/cli` and `automagent` to npm
- Verify `npx automagent init --quick` works end-to-end

## 8. Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Model API costs during `run` | Low | User provides own API key; we never store or proxy |
| Import mappings incomplete | Medium | Flag unmapped fields with TODO comments; iterate with design partners |
| Schema changes from partner feedback | Medium | Schema already has `additionalProperties: true`; additive changes only |
| "Why not just use Git?" objection | High | CLI commands (validate, run, import) prove value beyond static files |
| Framework landscape shifts | Medium | Framework-neutral core; framework-specific config in `extensions.*` only |
